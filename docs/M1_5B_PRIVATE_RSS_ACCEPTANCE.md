# M1.5B Private RSS Feed Acceptance

Status: implemented in current workspace changes.

## Scope

M1.5B adds revocable private RSS feeds and feed-token-backed enclosure access for authorized users.

Implemented:

- `rss_feeds` persistence with hashed opaque tokens, token prefixes, status, rotation, revocation, and last-used tracking.
- `program_access_grants` persistence for selected-user authorization checks.
- User RSS management APIs for listing, creating, rotating, revoking, and deleting feeds.
- Admin RSS list and revoke APIs.
- Private RSS XML endpoint that filters to currently authorized, published Programs and Episodes with published media.
- Feed-token-backed private enclosure URLs using the same opaque token.
- XML escaping, ETag, Last-Modified, and immediate token/access revocation enforcement.
- Mock user RSS management page at `/rss` with one-time link display, rotate, revoke, and delete flows.
- HTTP tests for invalid token, revoked token, ownership checks, token rotation invalidation, and enclosure range support.
- Playwright user test for create, rotate, and revoke behavior on the mock RSS page.

Not implemented:

- Public RSS.
- Payment or purchase flows.
- Real API-backed user catalog pages.
- Search or discovery over feeds.
- Storage of plaintext tokens after create or rotate.

## Feed Rules

- Tokens are generated by the server as opaque random strings.
- Only token hashes are stored.
- Plaintext tokens are returned only once on create or rotate.
- Token ownership never widens access beyond the current user's active Program grants.
- Revoked or expired feeds fail safely without distinguishing token existence from token state.
- Program access revocation immediately removes content from RSS and blocks enclosure delivery.
- RSS output excludes non-published Programs, non-published Episodes, and non-published media.

## API

- `GET /me/rss-feeds`
- `POST /me/rss-feeds`
- `POST /me/rss-feeds/{feedId}/rotate`
- `POST /me/rss-feeds/{feedId}/revoke`
- `DELETE /me/rss-feeds/{feedId}`
- `GET /admin/rss-feeds`
- `POST /admin/rss-feeds/{feedId}/revoke`
- `GET /admin/programs/{programId}/access-grants`
- `POST /admin/programs/{programId}/access-grants`
- `POST /admin/program-access/{grantId}/revoke`
- `GET /rss/private/{opaqueToken}.xml`
- `HEAD /rss/private/{opaqueToken}/episodes/{episodeId}/media`
- `GET /rss/private/{opaqueToken}/episodes/{episodeId}/media`

## Security Boundary

Private RSS responses do not expose:

- User email addresses.
- Internal user IDs.
- Storage keys.
- Artifact IDs.
- Connector or Job internals.
- Plaintext tokens outside one-time create or rotate responses.

Token-oriented responses enforce:

- `Referrer-Policy: no-referrer`
- `X-Robots-Tag: noindex, nofollow, noarchive`
- private no-store caching headers

The mock frontend page does not write tokens to:

- `localStorage`
- `sessionStorage`
- URL state
